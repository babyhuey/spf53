# spf53 — Design Spec (2026-07-20)

## What

`spf53` is an open-source, self-hosted SPF flattener for Route53. It solves the
RFC 7208 10-DNS-lookup limit the way hosted services (PowerSPF, AutoSPF) do —
resolve provider `include:` records, flatten to IP mechanisms, publish chained
TXT records — but as a small pip-installable tool you run yourself as a
scheduled Lambda. Tagline: *"spf53 — reapply your SPF automatically."*

Core loop (every 1h by default): resolve → flatten → diff against live DNS →
guard-check → atomic Route53 upsert → SNS notification. No diff, no writes, no
noise.

## Why

Static flattened SPF records go stale silently when providers change IPs
(dmarcian discontinued their flattening product for exactly this reason).
Hosted flatteners fix staleness but add a paid third-party dependency in the
mail path. spf53 is the ~500-line middle: automatic re-flattening with loud
alerting and refuse-to-apply safety guards.

## Decisions (settled with user)

- Name: **spf53** (SPF + port 53 + Route53). GitHub: `babyhuey/spf53`, public, MIT.
- Route53-only. DNS-update code isolated in one module so other backends could
  be added later; do NOT build abstraction layers now.
- Alerts: **SNS topic** only.
- Config: **SSM Parameter Store** (YAML string). CLI also accepts the same YAML
  from a local file via `-c/--config`.
- Packaging: proper Python package (`src/` layout, pyproject), CLI verbs
  `plan` / `apply` / `deploy`, plus `spf53.lambda_handler.lambda_handler`.
  Deployment is a boto3 bootstrap inside the tool — no SAM/CDK/Terraform.
- Python >= 3.11, runtime deps ONLY: `dnspython`, `boto3`, `pyyaml`.
  Lint: ruff. Tests: pytest + moto (Route53/SNS/SSM/IAM/Lambda mocking),
  DNS mocked at our resolver seam (no live DNS in tests).

## Config schema

One SSM parameter (default name `/spf53/config`), YAML:

```yaml
sns_topic_arn: arn:aws:sns:us-east-1:123456789012:spf53-alerts  # optional; no alerts if absent
resolver_ips: ["1.1.1.1", "8.8.8.8"]     # optional, defaults shown
domains:
  - name: example.com
    hosted_zone_id: Z123EXAMPLE           # required (no zone auto-discovery in v1)
    policy: "~all"                        # default "~all"; "-all" allowed
    max_shrink_pct: 30                    # default 30
    passthrough:                          # copied verbatim, never flattened, placed first
      - "exists:%{i}._spf.mta.salesforce.com"
    includes:                             # what gets recursively flattened
      - _spf.google.com
      - amazonses.com
```

## Record layout

- spf53 NEVER touches the apex/root TXT record (site verifications live there).
  It manages only `_spf53-1.<domain>`, `_spf53-2.<domain>`, … TXT records.
- The user sets the apex once, manually: `v=spf1 include:_spf53-1.example.com ~all`.
  `plan` output prints this expected apex record and warns if the live apex
  doesn't reference `_spf53-1.<domain>` (warning only, never an error).
- Chunk record content: `v=spf1 <passthrough…> <ip4/ip6 mechanisms…>` packed so
  each TXT record value is at most TWO strings of <= 255 chars each. Every
  chunk except the last ends with `include:_spf53-N+1.<domain>`; the last ends
  with the policy (`~all`). Passthrough mechanisms go at the front of chunk 1.
- Mechanisms are deterministic: ip4 before ip6, each sorted by
  `ipaddress.ip_network` sort key, CIDRs collapsed via
  `ipaddress.collapse_addresses` (v4 and v6 separately). Same inputs must
  produce byte-identical records (stable diffs).
- Stale chunks: if the new chain is shorter than the live one, surplus
  `_spf53-N` records are DELETEd in the same change batch.
- Lookup cost = chain length + 1 (apex include) + count of DNS-querying
  passthrough mechanisms (`exists:`, `include:`, `a`, `mx`, `ptr`). Warn (in
  plan output and SNS message) when cost > 9.

## Module contracts (exact signatures — implementers must match)

All modules in `src/spf53/`. Full type hints everywhere. No module imports
`cli` or `lambda_handler`. `route53`, `notify`, `ssm` are the only modules
that touch boto3; `resolver` is the only one that touches dnspython.

### resolver.py
```python
class ResolutionError(Exception): ...     # message names the include/domain that failed

MAX_DEPTH = 10  # includes nested deeper than this raise ResolutionError

def flatten(
    includes: Sequence[str],
    resolver_ips: Sequence[str],
) -> list[ipaddress.IPv4Network | ipaddress.IPv6Network]
```
- Recursively resolves each include's TXT SPF record; follows nested
  `include:`; resolves `a`/`a:host`/`mx`/`mx:host` to networks (the bare
  forms resolve against the record's own domain); collects `ip4:`/`ip6:`.
  Ignores `all` terms in provider records. `ptr` and `exists` inside provider
  records are ignored with a logged warning (cannot flatten).
- `a`/`mx` mechanisms accept the full RFC 7208 §5.3 dual-cidr-length suffix
  `[/ip4-cidr-length][//ip6-cidr-length]` (e.g. `a:host/24`, `a:host//64`,
  `a:host/24//64`, `a//48`). A single `/len` applies to the ip4-cidr-length
  ONLY — it never affects AAAA results, which default to /128 unless a
  `//len` is also given; A results default to /32 when no `/len` is given.
- Cycle-safe (seen-set). ANY failure (NXDOMAIN, timeout after retries, no SPF
  record found at an include, >MAX_DEPTH, malformed `ip4:`/`ip6:` CIDR
  literal, or an `a`/`mx` prefix length out of range for its address family)
  raises ResolutionError naming the offending token and domain — never
  return partial data. Timeout 5s, 2 tries per query, alternating resolver IPs.
- Returns collapsed, deterministically sorted networks (v4 first, then v6).
- Implementation seam for tests: module-level `_query_txt(name, resolver_ips) -> list[str]`
  monkeypatchable; all DNS goes through it.

### chunker.py
```python
MAX_TXT_STRING = 255
MAX_STRINGS_PER_RECORD = 2
DNS_QUERYING_MECHANISMS = ("include:", "exists:", "a", "mx", "ptr")

def build_records(
    domain: str,
    networks: Sequence[IPv4Network | IPv6Network],
    passthrough: Sequence[str],
    policy: str,
) -> dict[str, list[str]]
    # {"_spf53-1.example.com": ["v=spf1 ip4:... include:_spf53-2.example.com"], ...}
    # each value is the list of <=255-char strings for that ONE TXT record

def lookup_cost(records: dict[str, list[str]], passthrough: Sequence[str]) -> int

def to_route53_value(strings: list[str]) -> str      # '"str1" "str2"' quoted form
def from_route53_value(value: str) -> list[str]
```
- Networks render as `ip4:a.b.c.d/n` / `ip6:.../n`; host routes render WITHOUT
  the prefix (`ip4:1.2.3.4`, not `/32`; ip6 without `/128`).
- Splitting a mechanism across the two strings of a record is forbidden;
  strings concatenate with no separator per RFC 7208 3.3, so the split point
  must fall on a mechanism boundary and a single space leads the second string.

### guards.py
```python
@dataclass(frozen=True)
class GuardResult:
    ok: bool
    reasons: tuple[str, ...]      # human-readable refusal reasons, empty when ok

def check_guards(
    live_networks: Sequence[...],   # parsed from live _spf53-* records; empty on first run
    new_networks: Sequence[...],
    max_shrink_pct: int,
) -> GuardResult
```
- Refuses when: new set is empty; or live is non-empty and total address count
  (sum of `network.num_addresses`) shrank by more than `max_shrink_pct` percent.
  First run (no live records) passes as long as new set is non-empty.

### route53.py
```python
def get_txt_records(zone_id: str, domain: str) -> tuple[dict[str, list[str]], dict[str, int]]
    # all existing _spf53-N.<domain> TXT records (via list_resource_record_sets,
    # paginated) plus the apex TXT record under key domain itself; values decoded
    # via chunker.from_route53_value. Second dict is each record's live TTL,
    # keyed by the same record name — Route53 DELETE requires an exact TTL
    # match, so callers need it to build a correct delete.
def apply_changes(
    zone_id: str,
    upserts: dict[str, list[str]],
    deletes: dict[str, list[str]],    # name -> current strings (needed for DELETE)
    delete_ttls: dict[str, int] | None = None,  # name -> live TTL; missing name falls back to `ttl`
    ttl: int = 300,
) -> None                              # ONE atomic change_resource_record_sets call
```

### core.py  (orchestration; only module that imports resolver+chunker+guards+route53+notify)
```python
@dataclass(frozen=True)
class DomainPlan:
    domain: str
    zone_id: str
    desired: dict[str, list[str]]
    live: dict[str, list[str]]          # only _spf53-* records
    upserts: dict[str, list[str]]       # records that differ or are new
    deletes: dict[str, list[str]]       # stale _spf53-N chunks
    guard: GuardResult
    lookup_cost: int
    apex_warning: str | None            # set when live apex doesn't reference _spf53-1,
                                         # including when the apex is absent entirely
                                         # (never an error, always just a warning)
    summary: str                        # human-readable: ranges added/removed vs live
    delete_ttls: dict[str, int] = {}    # live TTL per stale record in `deletes`, for apply_changes

    @property
    def has_changes(self) -> bool

@dataclass(frozen=True)
class RunResult:
    plans: tuple[DomainPlan, ...]
    errors: tuple[str, ...]             # per-domain messages: ResolutionError, a corrupt
                                         # ip4:/ip6: token in a live record, or (after apply)
                                         # a Route53 ClientError

def plan(cfg: Spf53Config) -> RunResult
    # a ResolutionError for one domain goes to errors and does not stop others;
    # same for a corrupt ip4:/ip6: token found in that domain's live records
def apply(cfg: Spf53Config, force: bool = False) -> RunResult
    # per domain: skip if no changes; refuse (SNS alert) if guard fails and not
    # force; else apply_changes + SNS summary. Resolution/token errors always
    # alert. A route53.apply_changes botocore ClientError for one domain is
    # caught, added to errors, alerted via notify.publish, and does not stop
    # the remaining domains.
```

### cli.py
```python
def main(argv: Sequence[str] | None = None) -> int
```
- argparse, prog `spf53`, subcommands:
  - `spf53 plan  [-c FILE | --ssm-param NAME]` → print per-domain diff +
    lookup cost + expected apex record. Exit 0 = no changes, 2 = changes
    pending, 1 = any error.
  - `spf53 apply [-c FILE | --ssm-param NAME] [--force]` → exit 0 on success
    (incl. no-op), 1 on any error or guard refusal.
  - `spf53 deploy -c FILE [--schedule "rate(1 hour)"] [--create-topic NAME]
    [--param-name NAME] [--function-name spf53] [--region ...]`
- Default config source when neither flag given: SSM `/spf53/config`.
- Entry point in pyproject: `spf53 = "spf53.cli:main"`.

### lambda_handler.py

def lambda_handler(event: dict, context: object) -> dict

### deploy.py  (idempotent bootstrap, called from cli)
Steps, each create-or-update, safe to re-run:
1. Optionally create SNS topic (`--create-topic`), else use ARN from config.
2. Validate + push local YAML to SSM param.
3. IAM role `spf53-lambda` with inline least-privilege policy:
   `route53:ChangeResourceRecordSets`+`ListResourceRecordSets` on the
   configured zone ARNs only; `ssm:GetParameter` on the param ARN;
   `sns:Publish` on the topic ARN; CloudWatch Logs create/put.
4. Zip build: `pip install --target` runtime deps + `src/spf53` into a temp
   dir (use tempfile, not /tmp literals), zip, create-or-update Lambda
   (python3.13, 256MB, 60s, handler `spf53.lambda_handler.lambda_handler`,
   env `SPF53_PARAM`).
5. EventBridge rule `spf53-schedule` (default `rate(1 hour)`) → Lambda +
   lambda add-permission (idempotent, catch ResourceConflictException).
Prints each step's outcome; `--dry-run` prints without doing.

## Testing requirements

- `tests/` with pytest; no network, no real AWS. moto for route53/sns/ssm/iam/
  lambda where practical; monkeypatch `resolver._query_txt` for DNS.
- Must-cover cases:
  - resolver: nested includes, cycle, depth limit, a/mx forms, failure raises
    (no partial data), deterministic order, CIDR collapse.
  - chunker: round-trip to/from route53 values; a mechanism never split across
    strings; chain integrity (every record N links to N+1, last has policy,
    only chunk 1 carries passthrough); 255-boundary edge (mechanism exactly at
    limit); many-networks case forcing >= 3 chunks; lookup_cost counts chain +
    apex + exists-passthrough.
  - guards: empty-new refusal; shrink > threshold refusal; shrink == threshold
    passes; first-run pass; grow passes.
  - core: no-op when live == desired; upserts+deletes computed correctly when
    chain shrinks; guard refusal produces no route53 calls; one domain's
    resolution error doesn't block the other domain.
  - cli: plan exit codes 0/2; apply --force overrides guard refusal.
  - config: minimal valid YAML; missing hosted_zone_id error; bad policy error.
- GitHub Actions CI (`.github/workflows/ci.yml`): push+PR, matrix python
  3.11/3.12/3.13: `ruff check`, `ruff format --check`, `pytest`.
